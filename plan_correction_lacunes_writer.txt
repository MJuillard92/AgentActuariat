# PLAN DE CORRECTION: Trois Lacunes Critiques du WriterAgent

## RÉSUMÉ EXÉCUTIF

Le WriterAgent actuel viole le pattern PURE en **trois points essentiels**:
1. **Omission silencieuse** de sections quand les données manquent
2. **Troncature silencieuse** quand les plages d'âges sont incomplètes
3. **Hallucination LLM** : GPT-4o peut inventer des chiffres

**Impact**: Le rapport peut paraître complet mais contenir des trous ou des inexactitudes, sans alerter l'utilisateur.

**Solution**: Trois mécanismes de validation qui **échouent explicitement** plutôt que de contourner le problème.

---

## LACUNE #1: OMISSION SILENCIEUSE

### État actuel (INCORRECT)

```
WriterAgent cherche {{ chi_squared_p }} dans data_store
  ├─ Si trouvé: affiche le tableau
  └─ Si absent: omet la section SANS warning
     └─ Utilisateur reçoit un rapport incomplet sans le savoir
```

### Problème

- Violation du contrat YAML (section requise)
- Pas de trace d'audit de ce qui manque
- Impossibilité pour MasterAgent de détecter et corriger

### Comportement PURE (CORRECT)

```
WriterAgent cherche {{ chi_squared_p }} dans data_store
  ├─ Si trouvé: affiche le tableau
  └─ Si absent: LÈVE WriterError("chi_squared_p manquant")
     └─ MasterAgent reçoit l'erreur et agit
```

### Plan de correction

**Étape 1: Identifier TOUS les champs "requis" du data_store**

Examiner le YAML:
- Pour chaque section du `processing_sequence`
- Lister tous les `inputs_required` explicites
- Lister tous les champs nécessaires pour les `tool_calls` (graph specs, table specs)
- Créer une liste : "REQUIRED_FIELDS" qui énumère TOUS les champs qui DOIVENT être présents

Exemple:
```
REQUIRED_FIELDS = {
    "section_2": ["observation_period_years", "exposure_by_age_male", "deaths_by_age_female", ...],
    "section_4.1": ["modeled_deaths_by_age", "ci_lower_by_age", "ci_upper_by_age", ...],
    "section_4.3": ["discount_by_age", "logit_slope", "logit_intercept", ...]
}
```

**Étape 2: Valider la présence de TOUS les champs requis**

Dans `_initialize_context()`:
- Comparer data_store avec REQUIRED_FIELDS
- Pour chaque champ requis:
  - Si présent: OK
  - Si absent: ADD à missing_fields list
- Si missing_fields non vide: LEVER WriterError(missing_fields, "data_store")

**Étape 3: Valider que les valeurs ne sont pas vides**

Même si le champ existe:
- Vérifier que la valeur n'est pas None, [], {}, ou ""
- Si vide: ADD à missing_fields
- Si missing_fields non vide: LEVER WriterError

**Résultat**: Pas de section omise silencieusement. MasterAgent sait exactement ce qui manque.

## LACUNE #2: TRONCATURE SILENCIEUSE

### État actuel (INCORRECT)

```
WriterAgent génère table avec âges 20-80
  └─ YAML demande âges 0-100
     └─ Rapport affiche une table tronquée SANS indication
        └─ Utilisateur ne sait pas qu'il manque 40 lignes
```

### Problème

- Contrat YAML violé (plage d'âges incomplète)
- Pas de trace que les données sont partielles
- Risque réglementaire (table incomplète acceptée par erreur)

### Comportement PURE (CORRECT)

```
WriterAgent génère table avec âges 20-80
  └─ YAML demande cohort_min_age=0, cohort_max_age=100
     ├─ Valider: data_store contient TOUS les âges 0-100
     ├─ Si absent même un âge: LEVER WriterError
     └─ MasterAgent demande à BuilderAgent de combler les trous
```

### Plan de correction

**Étape 1: Extraire les contraintes de plage du YAML**

Pour chaque section avec un graphique ou table par âge:
- Lire cohort_min_age et cohort_max_age du template
- Lire la spécification du graphique/table
  - Exemple: `axes.x.range: [0, "{{ cohort_max_age }}"]`
- Créer une liste: "REQUIRED_AGE_RANGES" 
  - Clé: nom du champ (e.g., "exposure_by_age_male")
  - Valeur: range requis (e.g., [0, 120])

**Étape 2: Valider la complétude des vecteurs d'âges**

Avant de générer chaque graphique/table:
- Récupérer le vecteur d'âges du data_store (e.g., exposure_by_age_male)
- Récupérer la plage requise du template (e.g., [0, 120])
- Vérifier: TOUS les entiers de 0 à 120 sont présents dans le vecteur?
  - Si oui: continuer
  - Si non: LEVER WriterError(missing_ages, "data_store")
    - Inclure: quels âges manquent, quel champ, quelle section

**Étape 3: Valider les vecteurs après chaque graphique/table**

Dans `_execute_tool_call()` ou `_generate_graph()`:
- Après récupération des données du data_store
- Vérifier que le vecteur couvre la plage complète
- Si incomplet: LEVER WriterError IMMÉDIATEMENT
- Ne PAS générer la table/graph avec des données partielles

**Résultat**: Aucun graphique ou table tronqué. Erreur levée clairement. MasterAgent sait exactement quels âges manquent.

## LACUNE #3: HALLUCINATION LLM

### État actuel (INCORRECT)

```
WriterAgent envoie prompt à GPT-4o:
  "Rédige la section méthodologie. Voici les données: {exposition totale, taux lissé, ...}"
  
GPT-4o génère du texte:
  "Nous avons observé 15 435 exposés avec un taux moyen de 0,47%..."
  
Problème: Ces chiffres ne sont PAS dans data_store. GPT-4o les a inventés.
```

### Problème

- Hallucination LLM: chiffres non vérifiables
- Pas de validation que le texte généré correspond aux données réelles
- Risque réglementaire (rapport contient des chiffres faux)

### Comportement PURE (CORRECT)

```
WriterAgent prépare un payload pour GPT-4o:
  - UNIQUEMENT les chiffres du data_store
  - UNIQUEMENT dans la forme exacte stockée
  
WriterAgent envoie prompt strictement contraint:
  "Rédige l'interprétation. Tu dois citer EXACTEMENT ces valeurs: ..."
  
GPT-4o génère du texte en utilisant les valeurs autorisées
  
WriterAgent valide post-génération:
  - Scanne le texte généré
  - Pour chaque chiffre cité: vérifier qu'il est dans data_store
  - Si chiffre inconnu: REJECTER le texte et RELANCER GPT-4o
```

### Plan de correction

**Étape 1: Contrôler ce que GPT-4o reçoit**

Dans `NarrativeWriter.build_prompt()`:
- **NE PAS** envoyer un résumé textuel vague (risque d'hallucination)
- **Plutôt** envoyer une structure stricte:
  ```
  Available values for this section:
  - observation_period_years: [2007, 2008, 2009, 2010, 2011]
  - total_exposure_years: 500000.5
  - total_deaths: 3500
  - smoothing_algorithm: "whittaker_henderson"
  - baseline_regulatory_table: "THF 00/02"
  
  You may cite these values ONLY by their exact names.
  Do NOT invent any other numbers.
  ```

**Étape 2: Instruire GPT-4o de NE PAS extrapoler**

Dans le system prompt de NarrativeWriter:
- Ajouter explicitement: "You are NOT allowed to invent, estimate, or extrapolate any values."
- Ajouter: "You may ONLY use the values provided in the 'Available values' section."
- Ajouter: "If you need a value that is not listed, STOP and report the missing value."

**Étape 3: Valider post-génération**

Après que GPT-4o retourne le texte narratif:
- Créer un `NarrativeValidator`
- Scanner le texte pour extraire tous les nombres cités
- Pour chaque nombre: vérifier qu'il correspond à une valeur connue du data_store
- Si un nombre est "orphelin" (non trouvé): 
  - REJETER le texte
  - LOGGER: "GPT-4o a cité une valeur non autorisée"
  - RELANCER avec un prompt plus strict OU LEVER WriterError

Exemple de validation:
```
Texte généré: "Nous avons observé 500000.5 années-personne et 3500 décès."
Vérification:
  - 500000.5 → trouvé dans data_store.total_exposure_years ✓
  - 3500 → trouvé dans data_store.total_deaths ✓
Résultat: Texte accepté

Texte généré: "Nous avons observé 500000.5 années avec un ratio hommes/femmes de 63/37."
Vérification:
  - 500000.5 → trouvé ✓
  - 63/37 → non trouvé dans data_store ✗
Résultat: Texte rejeté, relancer GPT-4o
```

**Étape 4: Limiter la régénération**

- Si validation échoue, relancer avec prompt plus strict (max 2 retries)
- Si toujours invalide après 2 retries: LEVER WriterError("LLM hallucination detected")
- MasterAgent sait que le narratif ne peut pas être généré

**Résultat**: Aucune valeur inventée. Tous les chiffres du rapport sont traçables au data_store. Auditabilité garantie.

## TABLEAU RÉCAPITULATIF

| Lacune | État Actuel | PURE | Plan | Validation |
|--------|------------|------|------|-----------|
| **#1: Omission silencieuse** | Section omise si données manquent | WriterError levée | Lister REQUIRED_FIELDS, valider présence dans `_initialize_context()` | Aucun champ absent sans erreur |
| **#2: Troncature silencieuse** | Table/graph tronqué si âges incomplets | WriterError levée | Lister REQUIRED_AGE_RANGES, valider couverture avant chaque tool_call | Aucun graphique/table sans couverture complète |
| **#3: Hallucination LLM** | GPT-4o peut inventer des chiffres | Valeur autorisée + validation post-gen | Prompt contraint, NarrativeValidator, relancer si invalide | Tous les chiffres du rapport tracés au data_store |

---

## ORDRE D'IMPLÉMENTATION

1. **D'abord**: Lacune #1 (omission silencieuse)
   - Cela change `_initialize_context()` - le fondement
   - Les deux autres dépendent de cette structure

2. **Ensuite**: Lacune #2 (troncature silencieuse)
   - Cela protège chaque `tool_call` avant qu'il ne génère du contenu

3. **Enfin**: Lacune #3 (hallucination LLM)
   - Cela contrôle les outputs de NarrativeWriter

---

## CRITÈRES D'ACCEPTATION

Après correction, pour CHAQUE lacune:

### Lacune #1
- [ ] REQUIRED_FIELDS défini et exhaustif
- [ ] `_initialize_context()` valide tous les champs requis
- [ ] WriterError levée si un champ manque (pas omission silencieuse)
- [ ] Message d'erreur inclut le champ manquant, la section, et la source attendue

### Lacune #2
- [ ] REQUIRED_AGE_RANGES défini pour tous les champs vectorisés
- [ ] Validation avant CHAQUE tool_call (graph, table)
- [ ] WriterError levée si plage incomplète (pas troncature silencieuse)
- [ ] Message d'erreur inclut les âges manquants

### Lacune #3
- [ ] NarrativeValidator implémenté
- [ ] Prompt de GPT-4o inclut strictement "Available values"
- [ ] Post-génération: validation que tous les chiffres sont autorisés
- [ ] Relance automatique si hallucination détectée (max 2 retries)
- [ ] WriterError si toujours invalide après retries
